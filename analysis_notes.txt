LLM推理延迟正交试验 - 数据分析说明
课程：试验设计（期中作业）
================================================================

----------------------------------------------------------------
一、试验概况
----------------------------------------------------------------

试验对象：本地部署的大语言模型（LLM）推理服务
试验目的：考察量化精度、批处理大小、CPU负载、内存压力
          四个因子对推理延迟的影响规律
试验设计：L9(3^4) 正交表，每组合重复20次，共180次测量

因子设置：
  A - 量化精度：Q6K（6bit）/ Q4KM（4bit）/ Q2K（2bit）
  B - Batch大小：1 / 2 / 4
  C - CPU负载：10% / 40% / 70%
  D - 内存压力：0GB / 2GB / 4GB

响应变量：
  y1 - 单次推理延迟（ms），主响应，越低越好
  y2 - 组内变异系数CV（%），稳定性指标，越低越好

----------------------------------------------------------------
二、描述统计（summary_stats.csv）
----------------------------------------------------------------

各组合均值延迟范围：2798 ms ~ 7411 ms，跨度约2.6倍

最低延迟：组合8（Q2K, B=2, C=10%, D=4G）均值 2798 ms
最高延迟：组合3（Q6K, B=4, C=70%, D=4G）均值 7411 ms

最稳定组合：组合1（Q6K, B=1, C=10%, D=0G）CV = 3.28%
            20次重复中最大值3736ms，最小值3314ms，
            波动极小，说明该条件下推理过程高度确定性

最不稳定组合：组合2（Q6K, B=2, C=40%, D=2G）CV = 33.7%
             原始数据显示前8次延迟约3400~3700ms（正常），
             第9次起突然跳升至4300ms、8100ms、9400ms，
             判断是CPU+内存双重压力触发了系统级资源竞争，
             导致延迟出现双峰分布（正常态 vs 抖动态）

注：y2_cv_pct列在results.csv中对每个组合是固定值，
    这是因为CV是组合层面的统计量，在数据采集阶段预计算。

----------------------------------------------------------------
三、多因素ANOVA（方法1）
----------------------------------------------------------------

模型：y1 ~ A + B + C + D（主效应模型，无交互项）
样本：180条原始记录

方差分析结果：
  所有四个因子均在 p < 0.001 水平显著（*** 显著）

效应量η²（各因子解释的总变异比例）：
  A 量化精度  36.6%  ← 最关键因子
  C CPU负载   17.7%  ← 第二重要
  B Batch大小  9.6%  ← 第三
  D 内存压力   6.4%  ← 第四
  残差        29.7%  ← 未解释部分，含交互效应

解读：
  量化精度单独解释了超过三分之一的延迟变异。
  这符合物理直觉：低比特量化减少了浮点运算量，
  计算密集型推理对精度格式极为敏感。

  残差高达29.7%，提示因子间存在交互效应，
  主效应模型无法完全描述数据。后续分析（克里金模型的
  θ参数估计）也支持这一判断。

Tukey HSD事后检验（因子A，两两比较）：
  Q2K vs Q6K：均值差 -2317ms，p < 0.001
              Q2K比Q6K平均快约2317ms（降幅42%）
  Q2K vs Q4KM：均值差 -1852ms，p < 0.001
  Q4KM vs Q6K：均值差 -465ms，p = 0.018
              三个水平两两均有显著差异

LOO（留一法）交叉验证精度：
  RMSE = 2352 ms，MAE = 2059 ms
  精度较低的原因：正交表LOO时某些因子水平在训练集中
  缺失，导致预测退化。ANOVA的设计目标是假设检验，
  不适合用于预测性能评估。

----------------------------------------------------------------
三补充、C×D 交互效应分析（cd_interaction.png）
----------------------------------------------------------------

为什么单独分析C×D：
  CPU负载（C）和内存压力（D）在操作系统层面存在
  物理耦合机制：
    高CPU负载 → 进程调度频繁 → 内存访问模式被打断
    高内存压力 → 触发swap → CPU额外处理IO中断
  两者叠加时，延迟抖动可能产生非线性放大。
  原始数据中组合2（C=40%,D=2G）CV=33.7%已给出强信号。

含C×D交互项的ANOVA结果：
  若 p < 0.05：证实交互效应显著，主效应模型不完整，
               建议在工程部署中将C和D作为联合约束来管理
  若 p ≥ 0.05：交互效应在当前数据下未达显著，
               可能因正交表点数有限、统计功效不足

交互效应图解读方法（cd_interaction.png，共三图）：
  左上 折线图：
    - 三条线若平行 → C和D独立作用，无交互
    - 三条线斜率不同或交叉 → 存在交互效应
    - 预期：D=4G（红线）斜率最陡，说明高内存压力下
      CPU负载对延迟的影响被放大

  右上 热力图：
    - 颜色越红表示CV越高（越不稳定）
    - 左下角（C=10%,D=0G）应为最深蓝（最稳定）
    - 右上角（C=70%,D=4G）应为最深红（最不稳定）

  下方 箱线图：
    - 展示原始数据分布，可直观看到尾部离群值
    - 若某个C×D组合箱体明显更宽，说明该条件下
      系统行为最不可预测

四、多元线性回归（方法2）
----------------------------------------------------------------

模型：y1_mean ~ A_s + B_s + C_s + D_s（标准化自变量）
样本：9个组合均值

拟合方程：
  延迟(ms) = 4481.4
             - 1003.2 × A'（量化精度，编码越大延迟越低）
             +  530.9 × B'（Batch越大延迟越高）
             +  734.3 × C'（CPU负载越高延迟越高）
             +  413.2 × D'（内存压力越高延迟越高）

模型质量：
  R² = 0.923，调整R² = 0.847
  整体F检验 p = 0.017，模型显著

显著性（各系数）：
  A（量化精度）：p = 0.008 **    显著
  C（CPU负载）：p = 0.023 *     显著
  B（Batch）：  p = 0.060 .     边缘显著
  D（内存压力）：p = 0.113      不显著（样本量仅9点所致）

  注意：D在ANOVA中显著（n=180），在回归中不显著（n=9）
  这是样本量差异导致的，并非矛盾，D的效应真实存在
  但在9点回归中功效不足以检出。

模型诊断：
  Durbin-Watson检验：DW=1.369，p=0.133，无显著自相关
  Breusch-Pagan检验：BP=6.643，p=0.156，无显著异方差
  模型假设基本满足（但9点建模统计功效有限，结论需谨慎）

LOO交叉验证：
  RMSE = 885 ms，MAE = 735 ms
  三种方法中预测精度最高

----------------------------------------------------------------
五、克里金代理模型（方法3）
----------------------------------------------------------------

设置：Matérn 5/2核，普通克里金（常数趋势），估计nugget
输入：4维归一化坐标 [0,1]，输出：9点均值延迟

超参数估计结果（θ = 相关长度，越小表示变化越剧烈）：
  θ(A) = 0.048  ← 极小，量化精度方向响应面陡峭
  θ(B) = 1.688  ← 较大，Batch方向变化平缓
  θ(C) = 2.000  ← 达到上界，C方向变化最平缓
  θ(D) = 2.000  ← 达到上界，D方向变化最平缓

  C和D的θ收敛到参数上界，说明9个设计点不足以
  稳定辨识4维克里金超参数，模型在C、D方向欠辨识。

LOO交叉验证：
  RMSE = 1316 ms，MAE = 1012 ms
  理论上克里金适合小样本，但本案例中9点不足以支撑
  4维超参数估计，导致表现不如线性回归。
  若设计点增加到20点以上，克里金优势会更明显。

----------------------------------------------------------------
六、三种方法比较（method_comparison.csv）
----------------------------------------------------------------

方法         LOO-RMSE   LOO-MAE   预测新点  量化不确定性
ANOVA        2352 ms    2059 ms      否          否
线性回归      885 ms     735 ms      是         部分
克里金       1316 ms    1012 ms      是          是

结论：
  在本研究的9点正交设计下，线性回归综合表现最佳：
  预测精度高、系数可解释、计算成本低。
  
  克里金的不确定性量化功能有价值（可给出预测置信区间），
  但需要更多设计点（建议 ≥ 20）才能充分发挥优势。

  ANOVA的价值在于严格的假设检验（F检验+Tukey），
  而非预测。

七、TOPSIS综合优化（topsis_result.csv）
----------------------------------------------------------------

方法：
  双目标优化：y1均值延迟（权重0.6）+ y2 CV（权重0.4）
  两个目标均为成本型（越小越好）
  采用向量归一化 + 欧氏距离计算正负理想解距离

排名结果：
  #1  组合1   Q6K,  B=1, C=10%, D=0G    得分 0.896
  #2  组合7   Q2K,  B=1, C=70%, D=2G    得分 0.790
  #3  组合4   Q4KM, B=1, C=40%, D=4G    得分 0.667
  #4  组合5   Q4KM, B=2, C=70%, D=0G    得分 0.636
  #5  组合8   Q2K,  B=2, C=10%, D=4G    得分 0.540
  #6  组合6   Q4KM, B=4, C=10%, D=2G    得分 0.534
  #7  组合9   Q2K,  B=4, C=40%, D=0G    得分 0.521
  #8  组合3   Q6K,  B=4, C=70%, D=4G    得分 0.331
  #9  组合2   Q6K,  B=2, C=40%, D=2G    得分 0.284

最优组合分析（组合1）：
  - 均值延迟 3506 ms（不是最低，组合8的2798ms更低）
  - CV 仅 3.28%（9个组合中最低，稳定性最佳）
  - 综合得分 0.896，领先第2名约0.10分
  - 结论：Q6K量化在轻载（低CPU、零内存压力）环境下
    提供了延迟与稳定性的最佳平衡

第2名 vs 第1名的权衡（重要）：
  组合7（Q2K, B=1, C=70%, D=2G）绝对延迟更低（3035ms），
  比组合1低约13%。但CV=12.6%（约为组合1的4倍）。
  若应用场景对尾延迟（P99延迟）更敏感，选组合1更安全。
  若追求吞吐量/平均延迟，且能容忍偶发峰值，可选组合7。

末位原因分析：
  组合2（#9，Q6K, B=2, C=40%, D=2G）得分最低
  根本原因：CV高达33.7%，存在严重抖动
  直接原因：原始数据显示该组合后期出现延迟跳升
  （见results.csv combo=2，第9次起）
  工程建议：避免在CPU≥40% 且内存压力≥2GB同时存在
  的条件下运行批量推理任务

八、主效应图解读（main_effects.png）
----------------------------------------------------------------

量化精度（左上）：
  Q6K→Q4KM→Q2K，均值延迟从约5500ms降至3100ms
  趋势非线性：Q6K到Q4KM降幅小，Q4KM到Q2K降幅大
  物理解释：2bit量化大幅降低了矩阵乘法的计算量

Batch大小（右上）：
  1→2→4，延迟从约3900ms单调增至5500ms
  趋势近似线性
  物理解释：Batch增大意味着单次推理处理更多token，
  总计算量增加

CPU负载（左下）：
  10%→40%→70%，延迟从约3600ms增至5300ms
  在40%~70%区间斜率变大（资源竞争加剧）
  物理解释：CPU负载增高时，操作系统调度器对推理
  进程的资源分配减少，产生更多上下文切换

内存压力（右下）：
  0→2G→4G，延迟从约3600ms增至5200ms
  在2G→4G段斜率明显变大（非线性）
  物理解释：大内存压力可能触发内存交换（swap），
  导致延迟急剧上升，4GB压力时已接近触发swap阈值

----------------------------------------------------------------
九、局限性与改进建议
----------------------------------------------------------------

1. 正交表不含交互效应
   本研究采用主效应模型，ANOVA残差29.7%提示存在
   显著交互效应（如C×D的CPU-内存协同效应）。
   改进：增加试验点，采用全因子设计或中心复合设计

2. 克里金模型点数不足
   9点数据不足以辨识4维超参数，θ(C)和θ(D)
   均收敛到上界。改进：增加中间水平点，
   设计点达到25~30个以上

3. 异常值处理
   组合2、8、9存在明显的延迟跳变现象，
   本报告保留了所有原始数据。
   改进：区分"稳定态"和"抖动态"分别建模

4. 仅测试了单一模型
   不同参数量（7B vs 13B）的模型可能表现不同，
   结论的外部有效性有待验证
